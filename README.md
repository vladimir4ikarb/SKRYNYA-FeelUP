# FeelUP - Технічна документація та логіка системи (Актуальна версія)

## 1. Призначення системи

FeelUP - система обліку для студії аеродизайну / товарного складу, що поєднує:

* складський облік товарів та витратних матеріалів
* роботу з клієнтами
* оформлення замовлень
* облік закупівель
* резервування товарів
* контроль браку
* технічний облік гелію
* аудит дій користувачів

Система побудована так, щоб бути сумісною з Google Sheets, AppSheet та подальшим перенесенням у повноцінну БД.

## 2. Архітектурний принцип

Використовується модель Header + Line Items:

* `orders` + `orderItems`
* `purchases` + `purchaseItems`

Документ-заголовок зберігає загальні дані операції.
Рядки документа зберігають конкретні товари, кількість, ціну та суми.

Зв'язки між сутностями виконуються через ID.
Назви використовуються лише для інтерфейсу.

## 3. Основні сутності

### Товари (`products`)

Містять каталог позицій:

* назва
* категорія
* розмір
* колір
* ціна продажу
* собівартість
* норма гелію
* мінімальний залишок
* статус архіву / видалення

### Клієнти (`clients`)

Містять:

* ім'я
* телефон
* Instagram
* адреса
* коментар

### Замовлення (`orders`)

Заголовок продажу:

* клієнт
* дата доставки
* менеджер
* статус
* додаткові витрати
* загальна сума
* коментар

### Позиції замовлення (`orderItems`)

* товар
* кількість
* брак
* зафіксована ціна
* сума рядка

### Закупівлі (`purchases`)

Заголовок приходу:

* постачальник
* статус
* доставка
* загальна сума
* коментар

### Позиції закупівлі (`purchaseItems`)

* товар
* кількість
* собівартість
* сума рядка

### Логи (`logs`)

Фіксують:

* хто виконав дію
* тип дії
* старе значення
* нове значення
* час зміни

## 4. Складська логіка

Склад є розрахунковим джерелом істини.
Залишки не ведуться вручну.

### Формули

`Actual Stock = Прихід - Списання`

`Reserved Stock = усі qty у замовленнях зі статусом "В обробці"`

`Free Stock = Actual - Reserved`

### Що впливає на склад

#### Закупівлі (вхід)

Впливають статуси:

* Оплачено
* Борг

Не впливають:

* Чернетка
* Обробляється
* Скасовано

#### Замовлення (вихід)

На списання впливає:

* Виконано -> списання `qty + defect`

На резерв впливає:

* В обробці -> резерв `qty`

Не впливають:

* Чернетка
* Скасовано

## 5. Фінансова логіка

### Доход

Сума всіх виконаних замовлень.

### Собівартість

Береться з зафіксованої закупівельної ціни або актуальної `costPrice` залежно від режиму звіту.

### Маржа

`Продажі - Собівартість - Додаткові витрати`

## 6. Гелій

Для товарів може зберігатися норма витрати гелію.

При виконанні замовлення система може рахувати:

`(qty + defect) x heliumVolume`

Допускається окремий технічний облік балонів.

## 7. Цілісність даних

Критичні операції повинні виконуватись транзакційно:

* створення замовлення
* зміна статусу
* копіювання замовлення
* додавання позицій
* списання товару

Якщо кількість перевищує `Free Stock`, операція відхиляється.

## 8. Видалення та архів

Фізичне видалення робочих даних не використовується.

### Soft Delete

* `isDeleted = true`

### Archive

* `isArchived = true`

Товари та клієнти з історією операцій архівуються замість видалення.

## 9. Аудит лог

Колекція `logs` є immutable:

* дозволено створення записів
* редагування заборонене
* видалення заборонене

## 10. Legacy сумісність

Тимчасово допускаються поля:

* `delivery` -> `deliveryDate`
* `manager` -> `managerId`
* `supplier` -> `supplierName`
* `price` у закупівлі -> `costPrice`

Після стабілізації підлягають видаленню.

## 11. Принцип розвитку системи

Будь-які нові функції не повинні ламати:

* складські формули
  n- історичні дані
* логіку статусів
* аудит
* сумісність з існуючими документами

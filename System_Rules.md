# System Rules (Еталон архітектури системи)
**Версія:** 1.0 (Target State)

Цей документ описує фундаментальні принципи роботи FEEL UP CRM як єдиної екосистеми.

---

## 1. Принципи даних (Data First)
### 1.1 Firestore як Source of Truth
- Будь-які бізнес-дані (категорії, типи балонів, кольори, статуси, довідники) повинні зберігатися у Firestore.
- **Заборонено**: хардкодити списки вибору в TypeScript коді. Всі `SUB_CATEGORIES`, `COLORS`, `PRODUCERS` мають завантажуватися з колекції `/metadata/dictionaries`.

### 1.2 Структура документів
Всі сутності (товари, клієнти, замовлення) мають спільний мета-блок:
- `id`: string (UUID or Firestore ID)
- `createdAt`: serverTimestamp
- `updatedAt`: serverTimestamp
- `createdBy`: userId
- `isActive`: boolean (для soft delete)

## 2. Модульна архітектура
Система складається з незалежних, але пов'язаних модулів:
- **CRM**: Клієнти, Замовлення, Джерела продажів.
- **ERP (Inventory)**: Склад, Товарна матриця, Закупівлі, Списання.
- **Finance**: Каса, Витрати, Прибуток (автоматично на базі продажів та закупівель).
- **Admin**: Управління довідниками, права доступу, логування.

## 3. Статусна модель
Кожен документ (Замовлення, Закупівля) має чіткий життєвий цикл:
- **Draft**: Чорновик, не впливає на залишки та фінанси.
- **Confirmed**: Підтверджено (резервує товар).
- **Completed**: Виконано (списує залишок, створює фінансовий рух).
- **Cancelled**: Скасовано (знімає резерв).

## 4. Безпека та Аудит
### 4.1 Soft Delete
Дані ніколи не видаляються фізично, якщо вони мають зв'язки. Використовується прапор `isActive: false`.

### 4.2 Audit Log
Всі критичні зміни (зміна ціни, зміна залишку вручну, видалення замовлення) повинні реєструватися в колекції `/audit_logs` із зазначенням:
- Хто змінив.
- Що було до.
- Що стало після.
- Дата та час.

## 5. Принципи масштабування
- **Sub-collections vs Collections**: Використовувати підколекції для даних, що мають сенс тільки в контексті батька (н-клад: `orders/{id}/items`).
- **Denormalization**: Допускається дублювання назви товару в замовленні для збереження історії (статика на момент продажу), але ID товару має залишатися посиланням на актуальний об'єкт.

---

## 6. Заборони (Anti-patterns)
1. **Direct UI updates**: Заборонено міняти залишок товару напряму з UI без створення документа-підстави (Продаж, Списання, Закупівля).
2. **Missing handleFirestoreError**: Всі операції з БД повинні проходити через уніфікований обробник помилок для діагностики правил безпеки.
3. **Implicit logic**: Вся логіка розрахунків має бути винесена в сервіси (`services/`), а не жити всередині компонентів.
